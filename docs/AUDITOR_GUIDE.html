2. The Window Key — what a "window" actually is

A window = one continuous slice of game time (median ~43 seconds, whistle-ish boundaries, same manpower throughout — a goal or penalty always ENDS a window). Players can change on the fly inside one. The file gives one row per player per window — his view of that slice.

5. The six checks — plain English + the exact move

Check 1 · Source & lineage — "where did this number come from?"

Like tracing a steak to the farm. Use Section 4's "computed in" column, open that script, find the lines, confirm the chain reaches a rank-1/2/4 source. Move: fill "path verified ✓" or flag "can't trace" per feature.

Check 2 · Coverage — "is it filled in, and what happens when it isn't?"

python -c "import pandas as pd; d=pd.read_parquet(r'D:\optbot\artifacts\member_shells.parquet'); print(d.groupby(d.gamePk//1000000).apply(lambda g: g.form_prior_xgf60.isna().mean().round(3)))"

Blanks are often CORRECT (rookies have no prior seasons!) — the failure is a blank with no written rule. Move: record %-missing per season + the stated impute/flag policy. No policy = red flag.

Check 3 · Lag proof — "does it secretly know the future?"

An honest yesterday-stat predicts today and tomorrow EQUALLY (it can't know anything special about today). A leaky one predicts today way better. Script 13 measures exactly that, with a planted cheater as the control (the cheater screams ~0.9; honest columns sit near 0.00).

python scripts\13_lag_symmetry_audit.py --year 2022

Move: confirm every feature appears in the output with gap < 0.02; any new column you can't find there → extend the test to it (ask Claude for the 3-line pattern).

Check 4 · Legality — "could we know this BEFORE the prediction moment?"

The product is "pretend it's trade day — predict." So ask, per feature: if a GM asks about a move that hasn't happened, can we produce this value? Yes exactly (coach, schedule) → legal. Yes as a historical average (typical fatigue in that slot) → legal-as- distribution. Only knowable after play happens (shots in the window, when he left the ice) → BANNED. Move: your classification must match optbot/contracts/tower_schema.py; disagreements → escalate, don't silently fix — your disagreement might be the bug catch.

Check 5 · Stability — "does it behave the same every season?"

python -c "import pandas as pd; d=pd.read_parquet(r'D:\optbot\artifacts\perfect_windows_v3.parquet', columns=['season','team_xgf60_prior_ev']); print(d.groupby('season').team_xgf60_prior_ev.agg(['mean','std']).round(3))"

You want a boring flat-ish line. A sudden jump in one season is almost never hockey — it's a pipeline change (unit switch, broken join). Move: eyeball mean/std per season per feature; investigate any jump >~20% with no known cause.

Check 6 · Serving parity — "can the live product actually FETCH this at query time?"

Classic ML death: features that exist in training files but can't be computed when a customer clicks — the live model then runs on a different diet than it trained on. Move: for each feature, answer "when the demo asks about a hypothetical trade next week, where does this value come from?" If the answer is "…it can't" → flag train-only, keep out of the model. Sanity aid: scripts/04_project_trades.py is the serving path — a feature with no representation there deserves suspicion.
